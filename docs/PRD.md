# PRD: CUDA/C++ SPH Development Harness

## 목표
기존 CUDA/C++ 기반 SPH 코드를 지속적으로 확장하고, 논문에서 제안한 기법을 구현 및 검증할 수 있는 연구개발 하네스를 운영한다.

## 사용자
- 1차 사용자: 레포 소유자
- 2차 사용자: 소유자의 지시에 따라 구현과 검증을 수행하는 Codex

## 반복 작업 유형
1. 기존 SPH 코드에 새로운 기능 추가
2. 논문 기반 기법 구현 및 기존 코드에 통합
3. 수치 안정성, 정확도, 회귀 여부 검증
4. 케이스별 성능 및 결과 산출물 정리

## 대표 작업 클래스
### 1. 물리 모델 확장
- 예: 점성 모델 변경, 표면장력 추가, 경계조건 개선, EOS 교체
- 필요 입력: 대상 식 또는 모델 설명, 영향 받는 상태 변수, 검증 케이스
- 필수 출력: 구현 요약, 변경된 방정식 또는 가정, 기준 결과와의 비교

### 2. 수치 기법/안정화 기법 추가
- 예: time stepping 수정, density filtering, artificial viscosity 조정, correction term 추가
- 필요 입력: 수치 기법의 목적, 안정화 대상, 실패 모드
- 필수 출력: 안정성 개선 여부, 정확도 손실 여부, 기존 케이스 회귀 비교

### 3. 논문 재현 구현
- 예: 특정 SPH 논문의 pressure solver, boundary treatment, multiphase treatment 구현
- 필요 입력: 논문 출처, 핵심 식 번호, 알고리즘 절차, 재현 대상 표/그림
- 필수 출력: 논문 대비 차이점, 재현 수준, 기존 코드와의 통합 방식

### 4. 성능 개선
- 예: CUDA kernel memory access 개선, neighbor search 최적화, launch 구성 조정
- 필요 입력: 현재 병목, 기준 성능 측정값, 정확도 유지 조건
- 필수 출력: 성능 비교표, 정확도 회귀 없음 확인, 적용 범위

## 핵심 산출물
1. 기존 코드 구조를 해치지 않는 CUDA/C++ 구현
2. `CMake + CTest`로 재현 가능한 빌드 및 검증 절차
3. 기준 케이스 대비 수치 결과 비교 자료
4. 논문 재현 또는 기능 확장 결과를 설명하는 실험 기록
5. `docs/RESULTS_POLICY.md` 형식에 맞는 비교 가능 결과 산출물

## 성공 기준
- 새 기능 또는 논문 기법이 기존 SPH 베이스라인에 통합된다.
- 구현 후 최소 1개 이상의 기준 케이스로 정확도 또는 거동을 검증한다.
- 기존 검증 케이스와 비교해 명백한 회귀가 없어야 한다.
- 결과물은 다음 작업에서 재사용할 수 있도록 입력 조건, 출력 위치, 비교 기준이 문서화된다.
- 결과 산출물은 `docs/RESULTS_POLICY.md`의 기록 계약을 따른다.
- 비교 기준이 둘 이상이면 우선순위는 `docs/RESULTS_POLICY.md`를 따른다.

## 작업 접수 시 필수 정보
- 작업 유형: 기능 추가 / 논문 구현 / 회귀 조사 / 성능 최적화
- 변경 목표: 무엇을 바꾸려는지 한 문장으로 설명
- 근거 자료: 논문, 노트, 기존 코드 위치, 실패 로그 중 최소 하나
- 검증 기준: 어떤 케이스나 수치로 성공 여부를 판단할지
- 제약 조건: 유지해야 할 기존 기능, 성능 한계, 호환성 요구

## 기본 워크플로
1. 변경 대상 기능 또는 논문 기법을 정의한다.
2. 기존 SPH 베이스라인에서 영향을 받는 모듈과 데이터 흐름을 확인한다.
3. 구현 전에 검증 기준 케이스와 기대 결과를 정한다.
4. CUDA/C++ 코드와 필요한 테스트 또는 검증 로직을 추가한다.
5. `cmake`, `ctest`, 기준 시뮬레이션 실행으로 결과를 확인한다.
6. 수치 결과, 회귀 여부, 남은 한계를 정리한다.

## 작업 유형별 최소 검증
| 작업 유형 | 최소 검증 |
|-----------|-----------|
| 물리 모델 확장 | 빌드 + 관련 단위/통합 테스트 + 대표 케이스 1개 이상 |
| 수치 기법 추가 | 빌드 + 안정성 확인 케이스 + 기존 기준 케이스 회귀 비교 |
| 논문 구현 | 빌드 + 논문 재현 케이스 + 기존 베이스라인 비교 + 논문 대비 차이 기록 |
| 성능 개선 | 빌드 + 정확도 회귀 없음 확인 + 변경 전후 성능 비교 |

## 완료 정의
- 구현 목적과 변경 범위가 한 문단으로 요약되어 있다.
- 실행한 빌드/테스트/시뮬레이션 명령이 남아 있다.
- 결과 비교 기준과 실제 결과가 남아 있다.
- 남은 리스크, 미검증 항목, 후속 작업이 분리되어 기록되어 있다.

## 범위 제외
- 웹 UI, 대시보드, 서비스형 제품 기능
- 불필요한 프레임워크 도입이나 대규모 재작성
- 검증 기준 없이 커널 또는 물리 모델을 임의 확장하는 작업
- 논문 구현 시 참고 문헌과 기준 케이스 없이 결과만 맞추는 작업

## 문서화 원칙
- 논문 구현 작업은 출처, 핵심 식, 대상 모듈, 검증 케이스를 함께 남긴다.
- 기능 추가 작업은 기존 베이스라인 대비 무엇이 달라졌는지 명확히 적는다.
- 결과 요약은 정성적 설명만 쓰지 말고 가능한 한 수치 비교를 포함한다.
- 성능 개선 작업은 속도 향상 수치와 정확도 유지 여부를 함께 기록한다.
