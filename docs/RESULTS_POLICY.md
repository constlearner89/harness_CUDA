# 실험 결과 정책

## 목적
이 문서는 사용자 인터페이스 스타일 가이드가 아니라, SPH 시뮬레이션 결과와 검증 산출물을 일관되게 정리하기 위한 정책 문서다.

## 기본 원칙
1. 모든 실험 결과는 어떤 기능 또는 어떤 논문 기법을 검증하는지 바로 식별 가능해야 한다.
2. 결과 요약은 정성적 설명에 그치지 말고 가능한 한 수치 비교를 포함해야 한다.
3. 기존 베이스라인 대비 차이와 기대한 변화 방향을 함께 적어야 한다.
4. 재현에 필요한 입력 조건, 실행 명령, 출력 위치를 남겨야 한다.

## 반드시 남길 정보
| 항목 | 내용 |
|------|------|
| 실험 이름 | 기능명 또는 논문명 + 케이스명 |
| 목적 | 무엇을 검증하려는지 한 줄로 설명 |
| 입력 조건 | 입자 수, 시간 간격, 경계조건, 주요 상수 |
| 실행 명령 | 빌드/실행/검증에 사용한 명령 |
| 출력 위치 | 결과 파일, 로그, 이미지, 표 저장 경로 |
| 비교 기준 | 기존 베이스라인, 논문 수치, 기준 해 중 무엇과 비교했는지 |
| 결론 | 통과 여부, 남은 이슈, 추가 조사 필요 사항 |

## 결과 파일 작성 규칙
- 로그 파일은 케이스명과 날짜 또는 실행 식별자를 포함해 충돌을 피한다.
- 결과 디렉토리는 기능별 또는 논문별로 나누고, 기준 케이스와 변형 케이스를 함께 두지 않는다.
- 후처리 스크립트가 읽는 출력 포맷은 임의 변경하지 않는다.
- 그래프나 표를 만들었다면 어떤 원시 데이터에서 생성했는지 추적 가능해야 한다.

## 권장 디렉토리 예시
```text
results/
├── feature-name/
│   ├── case-a/
│   │   ├── run.log
│   │   ├── output/
│   │   ├── plots/
│   │   └── summary.md
│   └── case-b/
└── paper-name/
    ├── baseline/
    ├── reproduced/
    └── comparison/
```

## 권장 파일명 규칙
- 로그: `<case>-<variant>-run.log`
- 원시 결과: `<case>-<variant>-raw.csv` 또는 기존 포맷
- 비교 표: `<case>-comparison.md` 또는 `<case>-metrics.csv`
- 플롯: `<case>-<metric>.png`
- 요약: `summary.md`

## 권장 비교 방식
- 베이스라인 대비 오차 또는 차이값 표
- 시간에 따른 보존량 변화 그래프
- 논문 표/그래프와의 정성 또는 정량 비교
- 동일 케이스에서 변경 전후 결과 스냅샷 비교

## 비교 기준 우선순위
1. 기존 target-project baseline과의 비교
2. 논문 재현 작업이라면 논문 수치 또는 논문 그래프와의 비교
3. 기준 해 또는 reference solution과의 비교
4. 변경 전후 스냅샷 비교

- 여러 비교 기준이 동시에 있으면 위 우선순위를 따른다.
- 변경 전후 스냅샷 비교는 보조 자료로는 유효하지만, 명시적 예외가 없는 한 유일한 acceptance 근거로 쓰지 않는다.

## 논문 구현 전용 체크리스트
- 논문 제목, 저자, 연도, 식 번호가 기록되어 있는가?
- 논문에서 생략된 구현 디테일을 어떻게 해석했는지 적었는가?
- 논문 실험 조건과 현재 코드의 조건 차이를 기록했는가?
- 재현 목표가 정량 일치인지, 정성 경향 재현인지 명시했는가?
- 논문 결과와 다를 때 가능한 원인을 적었는가?

## 기능 추가 전용 체크리스트
- 기존 어떤 모듈을 확장했는지 적었는가?
- 기존 동작과 새 동작을 비교할 baseline이 있는가?
- 새 기능을 끄고 기존 동작으로 돌아갈 수 있는가?
- 검증 케이스가 기능 의도와 직접 연결되는가?

## 좋은 검증 요약의 형태
- 무엇을 바꿨는지
- 어떤 케이스로 검증했는지
- 기대한 결과가 무엇이었는지
- 실제 결과가 어땠는지
- 회귀가 있었는지
- 다음 단계에서 무엇을 더 확인해야 하는지

## 피해야 할 기록 방식
| 금지 사항 | 이유 |
|-----------|------|
| "결과가 좋아 보임" 같은 정성적 결론만 남기기 | 후속 검증과 비교가 불가능하다 |
| 입력 조건 없이 그래프만 저장하기 | 재현성이 없다 |
| 기준 베이스라인 없이 논문 구현 결과만 제시하기 | 개선인지 회귀인지 판단할 수 없다 |
| 실행 명령 없이 출력 파일만 남기기 | 같은 결과를 다시 만들 수 없다 |
| 성공 사례만 남기고 실패 케이스를 버리기 | 디버깅 근거와 설계 판단이 사라진다 |

## 최소 검증 패키지
- 빌드 명령
- 테스트 명령
- 대표 실행 명령
- 각 검증 명령의 실행 로그 경로
- 핵심 결과 파일 경로
- 비교 표 또는 비교 문장 1개 이상
- 남은 리스크 또는 미검증 항목

## Harness Executor 계약
- target-project validation step은 `steps/index.json`의 step 항목에 `results_contract`를 선언한다.
- `results_contract` 최소 필드:
  - `summary_path`
  - `output_paths`
  - `comparison_artifacts`
  - `comparison_basis`
  - `validation_log_paths`
- executor는 완료 직전 아래를 검증한다.
  - `summary_path` 파일 존재
  - `output_paths`, `comparison_artifacts`, `validation_log_paths`에 선언한 경로 존재
  - 요약 파일에 `실행 명령`, `실행 로그 위치`, `출력 위치`, `비교 기준`, `핵심 결과` 섹션 존재
  - 요약 파일의 `비교 기준` 항목에 `comparison_basis`가 기록되어 있음
  - `validation_commands`에 선언한 각 명령이 요약 파일과 validation log에 모두 기록되어 있음
