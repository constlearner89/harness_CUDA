# Architecture Decision Records

## 철학
이 프로젝트는 기존 CUDA/C++ 기반 SPH 코드의 점진적 확장과 검증 가능한 연구개발을 우선한다. 구현 속도보다 수치 정확성과 회귀 방지가 더 중요하며, 논문 구현도 기존 코드베이스에 유지보수 가능한 형태로 통합한다.

---

### ADR-001: CUDA/C++ 기반 SPH 베이스라인을 유지한다
**결정**: 새 기능과 논문 기법은 기존 CUDA/C++ SPH 코드베이스를 기준으로 추가한다.
**이유**: 현재 작업의 핵심 자산은 이미 작성된 SPH 베이스라인이며, 여기에 기능을 누적하는 방식이 가장 빠르고 검증 비용도 낮다.
**트레이드오프**: 초기 구조의 제약을 그대로 안고 가야 하며, 새로운 구현이 이상적인 구조가 아닐 수 있다.

### ADR-002: 기본 빌드 및 테스트 체계는 CMake + CTest로 고정한다
**결정**: 문서와 하네스 예시는 `cmake`와 `ctest`를 표준 빌드/검증 인터페이스로 사용한다.
**이유**: CUDA/C++ 프로젝트에서 가장 범용적이며, 로컬 개발과 자동화된 검증 흐름을 같은 방식으로 맞추기 쉽다.
**트레이드오프**: 외부 스크립트나 개별 실행 파일 중심의 즉흥적 검증 습관을 줄여야 하고, 테스트를 CTest 흐름에 맞추는 정리가 필요하다.

### ADR-003: 모든 구현은 검증 기준을 먼저 정한 뒤 진행한다
**결정**: 기능 추가와 논문 구현 모두, 구현 전에 기준 케이스와 기대 결과 또는 비교 대상을 먼저 정한다.
**이유**: 수치해석 코드는 컴파일 성공만으로 맞다고 볼 수 없고, 회귀 여부를 잡으려면 기준 데이터가 먼저 필요하다.
**트레이드오프**: 초기 구현 속도는 느려질 수 있지만, 디버깅 비용과 잘못된 결론 위험을 줄일 수 있다.

### ADR-004: 논문 구현은 일회성 실험 코드가 아니라 통합 가능한 모듈로 작성한다
**결정**: 논문에서 가져온 알고리즘도 가능한 한 `physics`, `core`, `neighbor`, `io` 등 기존 계층에 맞춰 통합한다.
**이유**: 이후 다른 논문 기법과 조합하거나 기존 기능과 비교할 때 유지보수성과 재사용성이 높아진다.
**트레이드오프**: 단기 재현만 목표로 할 때보다 설계 비용이 증가하고, 추상화 수준을 신중히 선택해야 한다.

### ADR-005: 기존 동작과 새 동작은 케이스 또는 설정으로 비교 가능해야 한다
**결정**: 새 기능이나 논문 구현은 가능하면 기존 베이스라인과 나란히 비교할 수 있도록 설정값, 실행 옵션, 케이스 분리 중 하나를 제공한다.
**이유**: 수치 코드에서는 “새 구현이 맞다”보다 “무엇이 얼마나 달라졌는가”를 비교 가능하게 만드는 것이 디버깅과 회귀 분석에 더 중요하다.
**트레이드오프**: 설정 분기나 비교용 케이스 관리가 늘어나며, 구현을 빠르게 단순 치환하는 방식보다 초기 비용이 든다.

### ADR-006: 결과 산출물은 재현 가능한 실험 단위로 남긴다
**결정**: 구현 결과는 실행 명령, 입력 조건, 출력 위치, 비교 기준이 함께 남는 실험 단위로 기록한다.
**이유**: 논문 구현과 기능 확장은 시간이 지난 뒤 다시 검증해야 하는 경우가 많고, 재현 불가능한 성공 기록은 가치가 낮다.
**트레이드오프**: 결과 정리 비용이 추가되며, 단기적인 실험 속도만 보면 번거로울 수 있다.
