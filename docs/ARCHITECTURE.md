# 아키텍처

## 목적
이 레포의 하네스는 CUDA/C++ 기반 SPH 전산해석 코드에 기능을 추가하거나 논문 기법을 구현할 때, 변경 범위를 통제하고 수치 검증까지 일관되게 수행하도록 돕는다.

## 권장 디렉토리 구조
```text
src/
├── core/               # 시간 적분, 파티클 상태, 공통 자료구조
├── kernels/            # CUDA 커널 및 저수준 GPU 연산
├── physics/            # 점성, 압력, 경계조건, 표면장력 등 물리 모델
├── neighbor/           # neighbor search, cell list, spatial hashing
├── io/                 # 입력 파싱, 설정, 결과 출력
├── cases/              # 대표 검증 케이스와 파라미터 세트
└── utils/              # 공통 수학/로그/헬퍼

tests/
├── unit/               # 수학 함수, 자료구조, CPU 유틸 단위 테스트
├── integration/        # solver 단계, 입출력, 모듈 결합 테스트
└── regression/         # 기준 케이스 결과 비교 및 회귀 검증

scripts/
└── ...                 # 빌드, 실행, 결과 비교 보조 스크립트
```

## 계층 및 책임
- `core/`는 시뮬레이션 루프와 상태 전이를 소유한다.
- `kernels/`는 GPU 병렬 연산만 담당하고, 상위 정책 결정은 넣지 않는다.
- `physics/`는 개별 물리 모델을 모듈 단위로 분리한다.
- `neighbor/`는 입자 탐색 구조를 독립적으로 유지해 물리 모델 변경과 결합을 낮춘다.
- `io/`는 케이스 설정, 파라미터, 결과 저장 형식을 담당한다.
- `tests/`는 단위 검증, 통합 검증, 수치 회귀 검증을 분리한다.

## 자주 바뀌는 변경 지점
- 압력, 점성, 표면장력, 경계조건 같은 물리 항은 `physics/`와 해당 CUDA 커널에서 함께 변경될 가능성이 높다.
- neighbor search 또는 cell list 변경은 `neighbor/`와 `kernels/`를 동시에 검토해야 한다.
- 시간 적분, CFL 조건, substep 처리 변경은 `core/`가 중심이지만 관련 출력과 회귀 기준도 함께 점검해야 한다.
- 결과 포맷 변경은 `io/`뿐 아니라 후처리 스크립트와 회귀 비교 방식에도 영향을 준다.

## 권장 모듈 경계
- 파티클 상태 정의와 메모리 배치는 `core/` 또는 전용 상태 모듈에 두고, 개별 물리 모델이 독자적으로 상태 구조를 소유하지 않는다.
- 논문 구현용 보조 함수는 먼저 해당 물리 또는 수치 모듈에 편입 가능한지 검토하고, 정말 필요할 때만 별도 하위 모듈로 분리한다.
- CUDA kernel은 계산에 집중하고, 파라미터 결정, 실험 스위치 선택, 출력 분기 같은 정책 로직은 호스트 코드에서 처리한다.
- 디버그/실험용 분기는 장기 유지가 필요한 기능인지 판단하고, 일회성이라면 케이스나 설정 레벨에 가둔다.

## 데이터 흐름
```text
케이스/설정 입력
→ 초기 입자 상태 및 시뮬레이션 파라미터 구성
→ neighbor search 구조 생성 또는 갱신
→ 물리 항 계산용 CUDA 커널 실행
→ 시간 적분 및 상태 업데이트
→ 결과 출력
→ 기준 해/기존 결과/논문 결과와 비교
→ 검증 보고서 또는 요약 생성
```

## 확장 패턴
- 새 기능은 기존 SPH 베이스라인을 보존하면서 필요한 모듈만 좁게 수정한다.
- 논문 구현은 논문 전용 실험 코드로 흩어 놓지 말고, 기존 아키텍처의 적절한 계층에 편입한다.
- GPU 커널과 호스트 측 오케스트레이션을 분리해 디버깅과 교체를 쉽게 만든다.
- 실험용 파라미터와 production-like 기본 설정을 혼동하지 않도록 케이스를 분리한다.

## 논문 구현 통합 절차
1. 논문의 핵심 기법이 물리 모델, 수치 기법, 데이터 구조 중 어디에 속하는지 먼저 분류한다.
2. 기존 베이스라인과 동일한 역할의 모듈이 있으면 그 확장으로 넣는다.
3. 기존 동작과 새 동작을 비교할 수 있도록 설정값 또는 케이스를 분리한다.
4. 논문 전용 예외 처리가 필요하더라도 solver 전체 분기를 만들기보다 국소적인 전략 전환을 우선한다.
5. 구현 후 논문 비교용 실험과 기존 회귀 케이스를 모두 실행한다.

## 구현 원칙
- CRITICAL: 검증 가능한 기준 케이스 없이 물리 모델 또는 수치 기법을 추가하지 않는다.
- CRITICAL: 기존 모듈을 대체하는 대규모 재작성보다 점진적 확장을 우선한다.
- 커널 최적화는 정확성 검증 이후에 진행한다.
- 논문 수식을 코드로 옮길 때는 식 번호, 변수 의미, 가정 조건을 추적 가능하게 남긴다.
- 출력 형식은 후속 비교 스크립트와 회귀 검증에 재사용 가능해야 한다.

## 변경 영향 체크리스트
- 이 변경이 파티클 상태 구조를 바꾸는가?
- neighbor search 또는 메모리 레이아웃에 영향이 있는가?
- 기존 케이스 결과 파일 포맷을 깨는가?
- 기존 물리 모델과 동시에 활성화될 수 있는가?
- 정확도 검증 외에 안정성 또는 성능 검증도 필요한가?
- 논문 구현이라면 논문과 구현 사이의 의도적 차이를 설명할 수 있는가?

## 빌드 및 검증 흐름
- 빌드 시스템 기본값은 `CMake`다.
- 자동 테스트 실행 기본값은 `CTest`다.
- repo root에 `src/`가 있으면 하네스는 이를 external target 후보로 간주한다. 루트 `CMakeLists.txt`가 없더라도 framework-only 분석으로 축소하지 않고, 먼저 최소 `CMake` + `CTest` 기준의 루트 `CMakeLists.txt`를 생성하는 bootstrap step을 계획해야 한다.
- bootstrap된 루트 `CMakeLists.txt`는 `src/`를 빌드 대상으로 포함하고, `tests/CMakeLists.txt`가 있으면 `enable_testing()`과 테스트 연결을 포함한다.
- 검증은 단위 테스트 통과만으로 끝나지 않고, 대표 시뮬레이션 케이스 실행과 결과 비교를 포함해야 한다.
- 결과 기록 형식과 비교 산출물은 `docs/RESULTS_POLICY.md`를 따른다.
- 비교 기준이 여러 개일 때의 우선순위도 `docs/RESULTS_POLICY.md`를 따른다.
- 하네스 step은 가능하면 다음 세 가지를 모두 갖는다:
  - 빌드 검증
  - 테스트 검증
  - 기준 케이스 실행 또는 결과 비교

## 권장 검증 명령 패턴
```bash
cmake -S . -B build
cmake --build build
ctest --test-dir build --output-on-failure
./build/<solver-binary> <case-config>
python3 scripts/<compare-script>.py <baseline> <candidate>
```
